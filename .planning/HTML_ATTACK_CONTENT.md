## VERSION ANGLAISE

```html
<h3>10 attack types modeled</h3>
<ol class='method-list'>

<li><strong>Brute force</strong> — Exhaustively try all possible character combinations until finding the correct password. Like testing all codes on a 4-digit lock: 0000, 0001, 0002... 9999. Mathematically, if a password has N characters and the character set has C possible characters (26 lowercase + 26 uppercase + 10 digits + ~32 symbols = 94), there are C^N possible combinations. Example: A 6-character password with 94 charset = 94^6 ≈ 689 billion combinations ≈ 5 minutes at 2,000 GH/s (MD5). An 8-character password = ~6 quadrillion combinations ≈ 95 hours. A 12-character password = 475 sextillion combinations ≈ millions of years. <strong>Why it works:</strong> Effective only on short passwords (&lt;8 chars). <strong>Protection:</strong> Use 12+ characters with mixed case, digits, and symbols. <a href='https://www.usenix.org/conference/usenixsecurity16/technical-sessions/presentation/wheeler' target='_blank' rel='noopener'>Source</a></li>

<li><strong>Dictionary</strong> — Test against ~14 billion known leaked passwords (HIBP database) plus dictionaries of common words. Humans create passwords based on real words due to memorability constraints. Instead of testing C^N combinations, attackers test only ~300,000 French words or ~1 million English words — reducible to seconds instead of years. Example: "soleil" is cracked in &lt;1 second; "password" cracks in milliseconds. <strong>Why so effective:</strong> 90% of real-world passwords are based on dictionary words because humans prioritize memorability. HIBP alone contains 14+ billion actual compromised passwords collected from real breaches. <strong>Statistics:</strong> The top 1,000 passwords (password, 123456, qwerty) account for ~20% of all passwords. <strong>Protection:</strong> Never use a real word alone. Use random generation or a passphrase of 5+ random words. <a href='https://haveibeenpwned.com/' target='_blank' rel='noopener'>Source</a></li>

<li><strong>Hybrid</strong> — Dictionary words + mutations (capitals, digits, symbols). Most mutations follow predictable patterns: adding digits to the end (password0-9), capitalizing first letter (Password), replacing letters with numbers (p4ssw0rd, e→3, s→5, o→0), reversing (drowssap). Tools like Hashcat document ~1,000 common mutation rules. One 300,000-word French dictionary × 1,000 mutations = 300 billion variants tested in hours. <strong>Real-world pattern:</strong> Someone thinks "I'll take a memorable word + add the year + add a symbol" and creates "Soleil2024!" — exactly what hybrid attack targets. This reflects actual human behavior. <strong>Effectiveness:</strong> Cracked in minutes to hours. <strong>Protection:</strong> Don't simply add mutations to dictionary words. Use random generation or distribute mutations randomly throughout instead of at the end. <a href='https://gist.github.com/Chick3nman/32e662a5bb63bc4f51b847bb422222fd' target='_blank' rel='noopener'>Source</a></li>

<li><strong>Mask</strong> — Target predictable human structures. Research shows 85% of passwords follow ~10 common patterns: [Uppercase][lowercase...][digits] (Password123), [lowercase...][digits] (password123), [Uppercase][lowercase...][digits][symbol] (Password1!), etc. Humans apply these patterns unconsciously. The math: A "perfect" password "Soleil2024" has 72 entropy bits (log2(94^10)) = 2^72 combinations in brute force ≈ 1 million years. Detected as pattern [Uppercase][lowercase 5-7][digits 4], it reduces to 26 × (26^5 to 26^7) × (10^4) ≈ 10^12 combinations ≈ 500 seconds. <strong>10 million times faster.</strong> <strong>Why:</strong> Humans follow cognitive/cultural patterns unconsciously. Uppercase at beginning "like real words", digits at end "tradition", symbols "for security". None of this is random. <strong>Protection:</strong> Randomly distribute case, digits, symbols throughout. Avoid predictable structure. <a href='https://www.usenix.org/conference/usenixsecurity16/technical-sessions/presentation/wheeler' target='_blank' rel='noopener'>Source</a></li>

<li><strong>Rainbow table</strong> — Pre-computed lookup table mapping password hashes to original passwords. Instead of computing hash(attempted_password) and comparing (slow), attackers pre-calculate hashes of billions of passwords once, then simply look up the stolen hash in the table (fast O(log n) lookup). Example: Stolen hash "5f4dcc3b5aa765d61d8327deb882cf99" → instant lookup → "password". <strong>Scale:</strong> Pre-computing hashes of 14 billion passwords (MD5 is fast) takes weeks on a GPU, creating a ~500 GB table. Once created, this table cracks any non-salted MD5/SHA-1/NTLM hash in milliseconds. <strong>Modern defense:</strong> Servers now use "salting" — adding a unique random value to each password before hashing. This breaks rainbow tables (each hash requires a different table). <strong>Vulnerable systems:</strong> Old Windows servers (unsalted NTLM), unsalted MD5/SHA-1. <strong>Protection:</strong> Modern servers use salting automatically. For users: avoid using common passwords (they're in tables), use unique/complex passwords unlikely to be pre-computed. <a href='https://www.hivesystems.com/blog/are-your-passwords-in-the-green' target='_blank' rel='noopener'>Source</a></li>

<li><strong>Credential stuffing</strong> — Reuse leaked email/password pairs from previous breaches against other services. When Facebook, Yahoo, LinkedIn are breached (billions of credentials leaked), attackers collect these email/password pairs. Since 80% of users reuse passwords across sites, these credentials work on Gmail, Twitter, banking, etc. 2-5% success rate is statistically guaranteed. Example: LinkedIn breach 2012 (6.5M credentials) + Yahoo 2017 (147M) + Equifax 2017 (147M) = hundreds of millions of reusable credentials. One breach = access to your Google, GitHub, Amazon, Slack if you reused. <strong>Why devastating:</strong> Completely outside your control. Your excellent password means nothing if a minor site you used is breached and you reused. <strong>Real impact:</strong> Studies show each 1 million credentials tested = 20,000-50,000 successful logins due to password reuse. <strong>Protection:</strong> Unique password per site (impossible to remember → use password manager like Bitwarden/1Password/KeePass), enable 2FA everywhere (blocks access even with correct password), check HIBP.com regularly. <a href='https://haveibeenpwned.com/' target='_blank' rel='noopener'>Source</a></li>

<li><strong>Password spraying</strong> — Test few common passwords (123456, password, admin, letmein) against millions of accounts instead of many passwords on one account. Strategically avoids anti-brute-force defenses (account lockout after N failed attempts). Attack pattern: 5,000 company employees, 10 common passwords tested = 50,000 attempts. If lockout triggers after 5 failed attempts per account, testing 10 passwords = 2 lockouts total across 5,000 accounts — defense rarely triggers. <strong>Success rate:</strong> ~1-2% of users choose a top-50 password = 50-100 accounts compromised per 5,000 employees. Enough for organizational breach. <strong>Scale:</strong> Used against Office365, Google Workspace, AWS. <strong>Example:</strong> Attacker tests "password" against 10M Gmail accounts across the internet. Even at 0.1% success rate = 10,000 accounts breached. <strong>Protection:</strong> Never choose a common password (even if "password" is forbidden, 123456 is used). Enable 2FA mandatory (negates password). Organization level: implement time-based lockouts (lock all logins from a pattern, not per-account), anomaly detection (alert on geographic/time anomalies). <a href='https://securelist.com/password-brute-force-time/112984/' target='_blank' rel='noopener'>Source</a></li>

<li><strong>Markov/probabilistic</strong> — Learn statistical patterns from real passwords, then prioritize likely sequences over random ones. Attackers analyze 1 billion real passwords and build a Markov chain: "Which character follows 'p'?" Answer: 'a' (15%), 'e' (12%), 'r' (10%), etc. The attack generates password candidates in order of statistical probability: "password" (extremely probable) → "passwor" (probable) → "passwork" (less probable) → "passwordx" (improbable) → "xyzabcd" (nearly impossible). <strong>Speed advantage:</strong> Tests statistically likely sequences first. Humans create passwords with linguistic patterns (common letter pairs like "qu", "ing", "able"). Machine-generated random passwords have no such patterns. <strong>Effectiveness:</strong> Cracks "natural" passwords 10-1,000× faster than brute force by prioritizing sequences with high real-world frequency. <strong>Limitation:</strong> Completely ineffective against truly random passwords (K7#mNpL!2xQvS9). <strong>Protection:</strong> Use machine-generated random passwords. Don't try to "create" a good password (you'll unconsciously include statistical patterns). Let a tool generate randomness. <a href='https://www.researchgate.net/publication/389902836' target='_blank' rel='noopener'>Source</a></li>

<li><strong>PCFG</strong> — Probabilistic Context-Free Grammar: learns grammatical password structures, not just individual characters. PCFG detects that 95% of human passwords follow ~20 common grammatical structures and tests matching passwords first. Learned structures: majuscules usually at start (40%), lowercase form continuous groups, digits at end (60%), symbols rare/at end. Real-world structures: [Uppercase][lowercase 5-7] (Password, Soleil), [Uppercase][lowercase 5-7][digits 4] (Password123, Soleil2024), [Uppercase][lowercase 5-7][digits 4][symbol] (Password1!). <strong>The power:</strong> Password "Soleil2024" (72 entropy bits, excellent in theory): Brute force = 2^72 ≈ 10^21 attempts ≈ 1 million years. Markov = ~10^14 attempts ≈ 50 hours. <strong>PCFG = ~10^11 attempts ≈ 1 second.</strong> 10 million times faster than brute force. <strong>Why:</strong> Humans create with unconscious logic: "majuscule for professionalism", "word ending with digits/symbols for security". PCFG learned these exact rules from billions of real passwords. <strong>Protection:</strong> Break grammatical structure. Don't create [Uppercase][lowercase][digits]. Create [lowercase][digit][Uppercase][symbol][lowercase][digit] (random distribution). Or use passphrase: correct-horse-battery-staple (no predictable grammar). <a href='https://www.usenix.org/conference/usenixsecurity16/technical-sessions/presentation/wheeler' target='_blank' rel='noopener'>Source</a></li>

<li><strong>Combinator</strong> — Concatenate two dictionaries to generate multi-word passphrases. Users create "soleilcheval" thinking "two dictionary words = very long = very safe". Attackers simply combine dictionaries: 300,000 French words × 10,000 common names = 3 billion combinations tested in hours. <strong>Examples:</strong> "soleil" + "cheval" = "soleilcheval", "correct" + "horse" = "correcthorse", "lune" + "tigre" = "lunetigre". Crackable. <strong>Scaling:</strong> 2 words = hours, 3 words = days, 4 words = months, 4 random words = millennia. <strong>Case study:</strong> xkcd's famous "correct-horse-battery-staple" is safe ONLY if words are truly random (1 in 10^15 chance). If words come from an ordered list ("correct" is word #1, "horse" is #50, etc.), a 4-dictionary combinator could theoretically test it. <strong>Real vulnerability:</strong> Two-word passphrases from common words (soleilcheval, correcthorse) are crackable. <strong>Protection:</strong> Use 4+ words (combinator-2 can't test combinator-4), add random separators (correct-horse!battery#staple@2024 breaks patterns), include random symbols/digits (correct-h0rse!battery#s7aple2024), or use pure random generation. 4-word passphrases remain excellent if words are truly random. <a href='https://www.researchgate.net/publication/389902836' target='_blank' rel='noopener'>Source</a></li>

</ol>
<h3>6 hashing algorithms</h3>
<p>Hashcat on 12× RTX 4090: MD5 ~2,000 GH/s, SHA-1 ~610 GH/s, SHA-256 ~272 GH/s, NTLM ~2,000 GH/s, bcrypt ~71 kH/s, Argon2id ~800 H/s.</p>
<h3>Limitations</h3>
<p>State/cloud attackers could be 10–1,000× faster. Phishing, keyloggers, SIM swapping bypass password strength entirely.</p>
```

## VERSION FRANÇAISE

```html
<h3>10 types d'attaques modélisés</h3>
<ol class='method-list'>

<li><strong>Force brute</strong> — Essayer exhaustivement toutes les combinaisons de caractères possibles jusqu'à trouver le bon mot de passe. Comme tester tous les codes d'un cadenas à 4 chiffres : 0000, 0001, 0002... 9999. Mathématiquement, si un mot de passe a N caractères et l'ensemble a C caractères possibles (26 minuscules + 26 majuscules + 10 chiffres + ~32 symboles = 94), il y a C^N combinaisons. Exemple : 6 caractères, 94 charset = 94^6 ≈ 689 milliards ≈ 5 minutes à 2 000 GH/s (MD5). 8 caractères = ~6 quadrillions ≈ 95 heures. 12 caractères = 475 sextillions ≈ millions d'années. <strong>Pourquoi ça marche :</strong> Efficace seulement sur mots courts (&lt;8 caractères). <strong>Protection :</strong> Utilisez 12+ caractères avec majuscules, minuscules, chiffres et symboles. <a href='https://www.usenix.org/conference/usenixsecurity16/technical-sessions/presentation/wheeler' target='_blank' rel='noopener'>Référence</a></li>

<li><strong>Dictionnaire</strong> — Tester contre ~14 milliards de mots de passe volés connus (base HIBP) plus dictionnaires de mots courants. Les humains créent des mots de passe basés sur des mots réels par nécessité de mémorisation. Au lieu de tester C^N combinaisons, les attaquants testent seulement ~300 000 mots français ou ~1 million anglais — réductible à secondes au lieu d'années. Exemple : "soleil" craqué en &lt;1 seconde ; "password" en millisecondes. <strong>Pourquoi très efficace :</strong> 90% des vrais mots de passe sont basés sur des mots de dictionnaire car les humains prioritaire la mémorisation. HIBP seul contient 14+ milliards de mots de passe compromis réels collectés de fuites authentiques. <strong>Statistiques :</strong> Les 1 000 mots de passe courants (password, 123456, qwerty) représentent ~20% de tous les mots de passe. <strong>Protection :</strong> Ne jamais utiliser un mot réel seul. Utilisez génération aléatoire ou passphrase de 5+ mots aléatoires. <a href='https://haveibeenpwned.com/' target='_blank' rel='noopener'>Référence</a></li>

<li><strong>Hybride</strong> — Mots dictionnaire + mutations (majuscules, chiffres, symboles). La plupart des mutations suivent des patterns prévisibles : ajouter chiffres à la fin (password0-9), capitaliser première lettre (Password), remplacer lettres par chiffres (p4ssw0rd, e→3, s→5, o→0), inverser (drowssap). Outils comme Hashcat documentent ~1 000 règles de mutation courantes. Un dictionnaire français 300 000 mots × 1 000 mutations = 300 milliards variantes testées en heures. <strong>Pattern réel :</strong> Quelqu'un pense "je vais prendre un mot mémorable + ajouter l'année + ajouter un symbole" et crée "Soleil2024!" — exactement ce que l'attaque hybride cible. Reflète comportement humain réel. <strong>Efficacité :</strong> Craqué en minutes à heures. <strong>Protection :</strong> N'ajoutez pas simplement mutations à mots dictionnaire. Utilisez génération aléatoire ou distribuez mutations aléatoirement partout au lieu de la fin. <a href='https://gist.github.com/Chick3nman/32e662a5bb63bc4f51b847bb422222fd' target='_blank' rel='noopener'>Référence</a></li>

<li><strong>Masque</strong> — Cible structures humaines prévisibles. Recherches montrent 85% des mots de passe suivent ~10 patterns courants : [Maj][min...][chiffres] (Password123), [min...][chiffres] (password123), [Maj][min...][chiffres][symbole] (Password1!), etc. Les humains appliquent ces patterns inconsciemment. Les maths : Un mot "parfait" "Soleil2024" a 72 bits entropie (log2(94^10)) = 2^72 combinaisons force brute ≈ 1 million années. Détecté comme pattern [Maj][min 5-7][chiffres 4], réduit à 26 × (26^5 à 26^7) × (10^4) ≈ 10^12 combinaisons ≈ 500 secondes. <strong>10 millions fois plus rapide.</strong> <strong>Pourquoi :</strong> Humains suivent patterns cognitifs/culturels inconsciemment. Majuscule début "comme mots réels", chiffres fin "tradition", symboles "pour sécurité". Rien n'est réellement aléatoire. <strong>Protection :</strong> Distribuez aléatoirement casse, chiffres, symboles partout. Évitez structure prévisible. <a href='https://www.usenix.org/conference/usenixsecurity16/technical-sessions/presentation/wheeler' target='_blank' rel='noopener'>Référence</a></li>

<li><strong>Table arc-en-ciel</strong> — Tableau pré-calculé géant mappant hashes de mots de passe aux originaux. Au lieu de calculer hash(mot_tenté) et comparer (lent), attaquants pré-calculent hashes de milliards de mots de passe une fois, puis cherchent simplement hash volé dans tableau (rapide recherche O(log n)). Exemple : Hash volé "5f4dcc3b5aa765d61d8327deb882cf99" → recherche instantanée → "password". <strong>Échelle :</strong> Pré-calculer hashes 14 milliards mots de passe (MD5 rapide) prend semaines sur GPU, créant table ~500 GB. Une fois créée, cette table craque n'importe quel hash non salé MD5/SHA-1/NTLM en millisecondes. <strong>Défense moderne :</strong> Serveurs maintenant utilisent "salting" — ajouter valeur aléatoire unique à chaque mot avant hachage. Casse tables arc-en-ciel (chaque hash requiert table différente). <strong>Systèmes vulnérables :</strong> Vieux serveurs Windows (NTLM non salé), MD5/SHA-1 non salés. <strong>Protection :</strong> Serveurs modernes salting automatique. Pour utilisateurs : éviter mots de passe courants (dans tables), utiliser mots de passe uniques/complexes peu probable pré-calculés. <a href='https://www.hivesystems.com/blog/are-your-passwords-in-the-green' target='_blank' rel='noopener'>Référence</a></li>

<li><strong>Credential stuffing</strong> — Réutiliser paires email/mot de passe volées de fuites antérieures contre autres services. Quand Facebook, Yahoo, LinkedIn piratés (milliards credentials volées), attaquants collectent ces paires email/mot de passe. Puisque 80% utilisateurs réutilisent mots de passe entre sites, ces credentials marchent sur Gmail, Twitter, banques, etc. Taux 2-5% réussite garanti statistiquement. Exemple : Fuite LinkedIn 2012 (6,5M credentials) + Yahoo 2017 (147M) + Equifax 2017 (147M) = centaines millions credentials réutilisables. Une fuite = accès Google, GitHub, Amazon, Slack si réutilisé. <strong>Pourquoi dévastateur :</strong> Complètement hors votre contrôle. Votre mot de passe excellent signifie rien si site mineur piraté et vous l'avez réutilisé. <strong>Impact réel :</strong> Études montrent 1 million credentials testés = 20 000-50 000 connexions réussies dues réutilisation. <strong>Protection :</strong> Mot de passe unique par site (impossible retenir → gestionnaire comme Bitwarden/1Password/KeePass), activer 2FA partout (bloque accès même mot correct), vérifier HIBP.com régulièrement. <a href='https://haveibeenpwned.com/' target='_blank' rel='noopener'>Référence</a></li>

<li><strong>Password spraying</strong> — Tester peu mots de passe courants (123456, password, admin, letmein) contre millions de comptes au lieu de beaucoup mots de passe sur un compte. Stratégiquement évite défenses anti-force-brute (verrouillage après N tentatives échouées). Pattern attaque : 5 000 employés entreprise, 10 mots de passe courants testés = 50 000 tentatives. Si verrouillage après 5 tentatives échouées par compte, tester 10 mots de passe = 2 verrous total à travers 5 000 comptes — défense rarement déclenche. <strong>Taux réussite :</strong> ~1-2% utilisateurs choisir top-50 mot de passe = 50-100 comptes compromis par 5 000 employés. Suffisant pour brèche organisationnelle. <strong>Échelle :</strong> Utilisé contre Office365, Google Workspace, AWS. <strong>Exemple :</strong> Attaquant teste "password" contre 10M comptes Gmail internet. Même 0,1% réussite = 10 000 comptes compromis. <strong>Protection :</strong> Ne jamais choisir mot de passe courant (même "password" interdit, 123456 utilisé). Activer 2FA obligatoire (annule mot de passe). Niveau organisation : implémenter verrouillages temps-réel (verrouiller tous logins d'un pattern, pas par-compte), détection anomalie (alerte géographie/temps anormaux). <a href='https://securelist.com/password-brute-force-time/112984/' target='_blank' rel='noopener'>Référence</a></li>

<li><strong>Markov/probabiliste</strong> — Apprendre patterns statistiques de vrais mots de passe, puis prioriser séquences probables plutôt qu'aléatoires. Attaquants analysent 1 milliard mots de passe réels et construisent chaîne Markov : "Quel caractère suit 'p'?" Réponse : 'a' (15%), 'e' (12%), 'r' (10%), etc. L'attaque génère candidats mots de passe ordre probabilité statistique : "password" (extrêmement probable) → "passwor" (probable) → "passwork" (moins probable) → "passwordx" (improbable) → "xyzabcd" (presque impossible). <strong>Avantage vitesse :</strong> Teste séquences statistiquement probables d'abord. Humains créent mots de passe avec patterns linguistiques (paires lettres courantes comme "qu", "ing", "able"). Mots de passe générés machine aléatoires pas ces patterns. <strong>Efficacité :</strong> Craque mots de passe "naturels" 10-1 000× plus rapide que force brute en priorisant séquences fréquence réelle élevée. <strong>Limitation :</strong> Complètement inefficace contre vrais mots de passe aléatoires (K7#mNpL!2xQvS9). <strong>Protection :</strong> Utilisez mots de passe générés machine aléatoires. N'essayez pas "créer" bon mot de passe (inconsciemment inclurez patterns statistiques). Laissez outil générer aléatoire. <a href='https://www.researchgate.net/publication/389902836' target='_blank' rel='noopener'>Référence</a></li>

<li><strong>PCFG</strong> — Grammaire contextuelle probabiliste : apprend structures grammaticales mots de passe, pas juste caractères individuels. PCFG détecte que 95% mots de passe humains suivent ~20 structures grammaticales courantes et teste mots de passe correspondants d'abord. Structures apprises : majuscules généralement début (40%), minuscules forment groupes continus, chiffres fin (60%), symboles rares/fin. Structures réelles : [Maj][min 5-7] (Password, Soleil), [Maj][min 5-7][chiffres 4] (Password123, Soleil2024), [Maj][min 5-7][chiffres 4][symbole] (Password1!). <strong>Le pouvoir :</strong> Mot de passe "Soleil2024" (72 bits entropie, excellent en théorie) : Force brute = 2^72 ≈ 10^21 tentatives ≈ 1 million années. Markov = ~10^14 tentatives ≈ 50 heures. <strong>PCFG = ~10^11 tentatives ≈ 1 seconde.</strong> 10 millions fois plus rapide que force brute. <strong>Pourquoi :</strong> Humains créent avec logique inconsciente : "majuscule pour professionnalisme", "fin mot de passe avec chiffres/symboles pour sécurité". PCFG appris ces règles exactes de milliards vrais mots de passe. <strong>Protection :</strong> Cassez structure grammaticale. Ne créez pas [Maj][min][chiffres]. Créez [min][chiffre][Maj][symbole][min][chiffre] (distribution aléatoire). Ou utilisez passphrase : correct-horse-battery-staple (aucune grammaire prévisible). <a href='https://www.usenix.org/conference/usenixsecurity16/technical-sessions/presentation/wheeler' target='_blank' rel='noopener'>Référence</a></li>

<li><strong>Combinatoire</strong> — Concaténer deux dictionnaires pour générer multi-mots passphrases. Utilisateurs créent "soleilcheval" pensant "deux mots dictionnaire = très long = très sûr". Attaquants simplement combinent dictionnaires : 300 000 mots français × 10 000 noms courants = 3 milliards combinaisons testées en heures. <strong>Exemples :</strong> "soleil" + "cheval" = "soleilcheval", "correct" + "horse" = "correcthorse", "lune" + "tigre" = "lunetigre". Crackables. <strong>Échelles :</strong> 2 mots = heures, 3 mots = jours, 4 mots = mois, 4 mots aléatoires = millénaires. <strong>Étude cas :</strong> Fameuse xkcd "correct-horse-battery-staple" sûre SEULEMENT si mots vraiment aléatoires (1 sur 10^15 chance). Si mots viennent liste ordonnée ("correct" mot #1, "horse" #50, etc.), attaque 4-dictionnaire combinatoire pouvait théoriquement tester. <strong>Vulnérabilité réelle :</strong> Passphrases deux mots mots courants (soleilcheval, correcthorse) crackables. <strong>Protection :</strong> Utilisez 4+ mots (combinatoire-2 peut pas tester combinatoire-4), ajoutez séparateurs aléatoires (correct-horse!battery#staple@2024 casse patterns), incluez symboles/chiffres aléatoires (correct-h0rse!battery#s7aple2024), ou utilisez génération pure aléatoire. Passphrases 4-mots restent excellentes si mots vraiment aléatoires. <a href='https://www.researchgate.net/publication/389902836' target='_blank' rel='noopener'>Référence</a></li>

</ol>
<h3>6 algorithmes de hachage</h3>
<p>Hashcat sur 12× RTX 4090 : MD5 ~2 000 GH/s, SHA-1 ~610 GH/s, SHA-256 ~272 GH/s, NTLM ~2 000 GH/s, bcrypt ~71 kH/s, Argon2id ~800 H/s.</p>
<h3>Limites</h3>
<p>Un attaquant étatique ou cloud pourrait être 10–1 000× plus rapide. Le phishing, les keyloggers et le SIM swapping contournent la force du mot de passe.</p>
```
